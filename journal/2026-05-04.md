# 2026-05-04

我发现我的情绪有一种微妙的连续性。比如我前一天想到了一个场景，第二天就发现这个情绪可以装进去。

有可能是因为我本身就一直在这些情结里，而我的内心在找出口。就会有一些场景。

想到了给写作云加一些机制打磨情绪。这个比较适合细节性的情绪化写作。虽然可以确认这对写营销文案有帮助，但是还不确认怎么迁移。

我清楚现在的写作云还只做了评审，还没有深入前面的创作流程。我知道距离最终成品还有比较远的距离。

想到了一些新的分析需求。比如分析 Opencode 的对话历史发现智能体工程的方向。再比如，发现 PRD 文档的共性和规律。

产品云是不是也可以从评审开始建立元规则。是个不错的主意哦。产品和资产一样也是抽象的概念，不像写作产出文章和书籍那样具体。这是 PaaS 区别于 SaaS 的核心特征之一。

产品云主要是帮助我看清自己的思路和方法，特别是一以贯之的核心思路。那我觉得可能发现高层次规律更重要。

产品云可以观察写作云。这样就可以互相观察来左脚踩右脚了。这里的难点在于确保迭代节奏不混乱。

验证产品研发方向对不对的核心方法是看有没有发现了从来没有发现的新现象，而又是符合我的内心的潜在认知的。

要不要把评审统一都定义成观测的一部分呢。这样方便迁移模式。

也就是所有的产品功能都从可观测性出发。这倒是一种与众不同的思路。大家都在聚焦于控制论。只有我在观测。

比如把写作评审改为观测，会有什么变化。评审再观测得到风格吗。那么评审这个概念又怎么用呢。

管理后台要和我的认知对齐。可以从一个静态网页开始做起。就当成一个内部的官网做，所有考虑项都记录到网站上，这样发布最快，发挥作用也最快。因为公开了大量资料，所以做起来也容易，甚至可以只有客户端。

各端都可以类似，比如资产云，也可以先把 GitHub 资产清单和契约写出来。

开源就是这点好。

多租户一直是一个心病。当前只用了一个存储桶。未来怎么隔离和演化还是麻烦事。多租户一旦多了也不好管理。看是需要 infra 入手还是 data 入手。感觉是 infra 的事情。
