🎯 各组件选型的战略价值

1. 服务端：FastAPI (Python)

· 为何选它：你对 Python 熟悉，且生态对数据和 AI 最友好。
· 关键特性利用：
  · Pydantic V2 作为全栈类型定义的唯一来源。
  · StreamingResponse 提升 AI 对话交互体验（流式输出）。
  · 自动 OpenAPI 文档 (/docs) 作为前后端契约，驱动代码自动生成。

2. 命令行客户端：Typer (Python)

· 为何是一等客户端：它不是运维脚本，而是面向终端用户（尤其是开发者/自动化场景）的独立产品界面。
· 战略价值：
  · 零成本复用：直接 import 服务端的业务逻辑模块和 Pydantic 模型。
  · 平滑演进：CLI 命令的核心函数，可以不加修改地被 FastAPI 路由调用，反之亦然。
  · 分发便捷：通过 pipx 安装，体验接近原生二进制，但保持了 Python 的开发效率。

3. GUI 客户端：Flutter (Dart)

· 为何先桌面后 PWA：
  · 开发效率：桌面端热重载最完整，调试环境最舒适。
  · 能力超集：桌面端可调用所有系统 API，Web 端作为其降级子集，向下兼容成本极低。
· 分发策略：
  · 桌面端：面向重度使用、需要本地文件访问等高级能力的用户。
  · PWA：面向非纯开发者（SaaS 普通用户），免安装、免商店、秒级更新，转化路径最短。
  · 移动端：当市场需要上架应用商店时，一键打包输出。

📦 范式总结陈述（可直接归档）

后端服务层 与 命令行客户端层 统一采用 Python 语言构建。其中 FastAPI 提供 RESTful / 流式 API 服务，作为数据与 AI 能力的中央网关；Typer 提供原生命令行交互界面，面向自动化及终端重度用户。两者共享 Pydantic 领域模型与核心业务逻辑代码，实现 ”CLI 功能即 API 雏形，API 端点即 CLI 复刻“ 的无缝双向演进。
图形化客户端层 统一采用 Flutter (Dart) 构建。开发阶段以 桌面端为第一目标平台，充分利用其热重载与系统集成能力进行快速迭代；发布阶段通过同一代码库顺势输出 PWA (Web 轻量入口) 及 iOS / Android 原生应用，实现 一次编写，四级覆盖 (桌面 / Web / iOS / Android)。
前后端协同 基于 OpenAPI 规范 自动生成类型安全的 Flutter 客户端代码，消除手写 API 调用与字段映射的维护负担。
整体架构 以 Python + Dart 双语言覆盖全终端场景，在保持技术栈简洁、开发心智负担低的前提下，兼顾了 轻量分发 (PWA)、原生体验 (桌面/移动) 与 自动化集成 (CLI) 的全面需求。
