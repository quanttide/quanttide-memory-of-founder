# 制衣厂数据清洗

## 行业背景

这是一个非常经典且核心的问题。直接回答你：这三个模块（工序、产量、员工）是制衣厂乃至整个离散制造业最通用的“核心三角”，但在具体的数据粒度和定义上，不同行业会有差异。
对于数据清洗项目来说，理解这三者背后的业务逻辑至关重要。下面我为你详细拆解一下这三个模块在制衣厂和制造业中的通用性，以及你在清洗数据时需要注意的“坑”。
一、 为什么说它是通用的？（核心逻辑）
在制造业管理中，有一个核心公式：
$$产值 = 员工 \times 工序 \times 产量$$
无论是制衣、电子组装、汽车配件还是玩具制造，只要是“离散制造”，都逃不开这三个要素：
员工： 谁干的？（关联考勤、计件工资、绩效）
工序： 干了什么？（关联工艺流程、工价、机器设备）
产量： 干了多少？（关联进度、交期、成本）
所以，这三个字段不仅通用，而且是MES（制造执行系统）和ERP（企业资源计划）中最基础的底层数据结构。
二、 制衣厂的特殊性（数据清洗的重点）
虽然逻辑通用，但制衣厂数据有其非常特殊的地方，这也是你做数据清洗项目的难点所在：
1. 工序信息 —— 最“乱”的数据
通用性： 所有工厂都有工序（如：裁剪、缝制、整烫、包装）。
制衣厂痛点：
非标准化： 制衣厂的工序拆分非常细，且不同款式的工序完全不同。做T恤和做羽绒服的工序天差地别。
别名多： 同一个工序，车间叫“埋夹”，系统里可能叫“合侧缝”，工人填单子可能写“车边”。清洗建议： 必须建立统一的【标准工序字典】，进行映射清洗。
工价关联： 工序数据里最关键的是“工价”（单价），这直接关系到钱。数据清洗时要检查是否有工序存在但无工价，或者工价异常（比如小数点点错）的情况。
2. 产量信息 —— 核心“钱”袋子
通用性： 所有工厂都统计产量。
制衣厂痛点：
计件制核心： 制衣厂绝大多数是计件工资。产量数据 = 工资数据。清洗建议： 产量数据容错率极低。你需要清洗出“负产量”（退货返工）、“重复录入”（多记账）的数据。
捆包制： 制衣厂常以“扎”或“包”为单位流转。数据里可能混合了“件数”和“包数”，清洗时需要统一单位。
返工与报废： 产量里往往包含“正品”和“次品”。制衣厂返工很频繁，如果不把返工次数单独清洗出来，会导致产量虚高，工资多算。
3. 员工信息 —— 相对标准
通用性： 工号、姓名、部门。
制衣厂痛点：
流动性极大： 制衣厂员工离职、入职非常频繁，甚至有大量临时工。数据清洗时常发现“产量表里有工号，但员工档案里查无此人”的情况。
多能工： 一个员工今天车位，明天可能去剪线。清洗时要注意员工技能标签的更新。

四、 给你的数据清洗项目的实操建议
既然你手头有这三个模块，建议你在清洗时增加或关联以下几个关键字段，能让项目价值翻倍：
时间维度：
产量表里一定要有“生产日期”甚至“班次”。制衣厂常有夜班，跨天数据容易混乱。
订单/款号：
产量必须关联“款号”。没有款号的产量是无效的，因为不同款的工价不同。
良品率/返工率：
单纯清洗产量是不够的，要尝试计算出“有效产量”。（例如：产量100件，返工20件，实际有效可能是80件，或者算作120次工作量，这取决于工厂规定）。
设备ID（机台号）：
虽然你目前没提到，但建议关联。制衣厂的设备（如平车、电脑车）效率差异很大。
总结：
这三个模块是绝对通用的底座。在制衣厂数据清洗中，最大的挑战不在于结构，而在于数据质量（录入不规范、变动频繁）。你的项目核心价值在于：把混乱的车间手写单据数据，转化为可分析的标准数据库。
