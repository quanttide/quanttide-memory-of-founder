# 代码转产品介绍资料 Skill

## 概述

本skill提供一套完整的工作流程，帮助您从项目的前后端**源代码**中提取关键信息，自动生成专业的、可直接发给客户的产品介绍资料。

## 核心约束：仅基于源代码生成

**⚠️ 本skill严格限定只读取和分析源代码文件，绝不读取任何非代码文件。**

### 允许读取的文件（源代码）

| 类别 | 文件扩展名 |
|------|-----------|
| JavaScript/TypeScript | `.js` `.jsx` `.ts` `.tsx` `.mjs` `.cjs` |
| Python | `.py` |
| Java/Kotlin | `.java` `.kt` |
| Go | `.go` |
| C/C++/C# | `.c` `.cpp` `.h` `.cs` |
| Rust | `.rs` |
| PHP | `.php` |
| Ruby | `.rb` |
| Swift/Objective-C | `.swift` `.m` |
| 前端模板 | `.vue` `.svelte` `.astro` |
| SQL | `.sql` |
| Prisma Schema | `.prisma` |

### 禁止读取的文件（非代码）

| 类别 | 文件/扩展名 | 原因 |
|------|------------|------|
| 文档 | `README.md` `CHANGELOG.md` `LICENSE` `*.md` `*.txt` | 属于人工描述，非代码事实 |
| 包管理 | `package.json` `package-lock.json` `yarn.lock` `pom.xml` `build.gradle` `requirements.txt` `Pipfile` `go.mod` `Cargo.toml` | 属于配置元数据，非代码逻辑 |
| 环境配置 | `.env` `.env.*` `config.*` `*.yaml` `*.yml` `*.toml` `*.ini` | 含敏感信息或部署配置 |
| 样式资源 | `*.css` `*.scss` `*.less` `*.sass` | 属于样式表现，非功能逻辑 |
| 静态资源 | `*.png` `*.jpg` `*.svg` `*.ico` `*.gif` `*.woff` `*.ttf` | 属于媒体资源 |
| 构建产物 | `dist/` `build/` `node_modules/` `.next/` `target/` | 属于编译输出 |
| Docker/CI | `Dockerfile` `docker-compose.*` `.github/` `Jenkinsfile` | 属于部署配置 |
| 数据文件 | `*.csv` `*.json`（非代码内的） `*.xml` | 属于数据文件 |
| 其他 | `.gitignore` `.eslintrc` `.prettierrc` `tsconfig.json` | 属于工具配置 |

### 分析原则

1. **产品名称**：从代码中的模块名、类名、命名空间推断，而非从 package.json 或 README 获取
2. **功能描述**：从代码逻辑（函数实现、路由定义、组件渲染）推断，而非从注释或文档获取
3. **技术栈**：从代码中实际使用的 import/require 语句推断，而非从依赖配置文件获取
4. **数据模型**：从代码中的类定义、类型定义、ORM模型推断，而非从数据库迁移文件获取

## 约束二：仅包含实际上线可用的功能

**⚠️ 代码中存在大量未上线、已下线或开发中的功能，必须严格过滤，确保产品资料中只包含实际上线可用的功能。**

### 必须排除的代码模式

#### 1. 被注释掉的代码

被注释掉的代码通常代表已下线、废弃或暂不可用的功能，**绝对不能**作为产品功能。

```bash
# 检测被注释掉的路由定义（仅代码文件）
grep -rn -E '^\s*(//|#)\s*(router|app)\.(get|post|put|delete)' --include="*.js" --include="*.ts" --include="*.py"

# 检测被注释掉的路由路径
grep -rn -E '^\s*(//|#)\s*path\s*[:=]\s*["\x27]' --include="*.tsx" --include="*.jsx" --include="*.vue"

# 检测被注释掉的组件引用
grep -rn -E '^\s*(//|#|<!--)\s*import.*from' --include="*.tsx" --include="*.vue" --include="*.js"
```

**处理规则：**
- 任何以注释符号（`//`、`#`、`<!-- -->`）开头的功能定义，一律排除
- 多行注释块（`/* */`、`""" """`）中包含的功能定义，一律排除
- 被条件编译指令包裹的代码（如 `#if 0` ... `#endif`），一律排除

#### 2. 开发中/待办标记

以下标记表明功能尚未完成或正在开发中，**不能**作为已上线功能：

| 标记 | 含义 | 处理 |
|------|------|------|
| `TODO` | 待办事项 | 排除 |
| `FIXME` | 需要修复 | 排除 |
| `HACK` | 临时方案 | 排除 |
| `XXX` | 需要关注的问题 | 排除 |
| `WIP` | 工作进行中 | 排除 |
| `IN PROGRESS` | 开发中 | 排除 |
| `DEPRECATED` | 已废弃 | 排除 |
| `@deprecated` | 已废弃（JSDoc/注解） | 排除 |
| `@removed` | 已移除 | 排除 |
| `@beta` | 测试阶段 | 标注为"即将推出" |
| `@experimental` | 实验性功能 | 标注为"即将推出" |
| `@alpha` | 早期阶段 | 排除 |

```bash
# 检测开发中标记（仅代码文件）
grep -rn -E '\b(TODO|FIXME|HACK|XXX|WIP|DEPRECATED)\b' --include="*.py" --include="*.js" --include="*.ts" --include="*.java" --include="*.vue" --include="*.tsx"

# 检测JSDoc/注解中的废弃标记
grep -rn -E '@(deprecated|removed|beta|experimental|alpha)\b' --include="*.py" --include="*.js" --include="*.ts" --include="*.java"
```

**处理规则：**
- 包含 `TODO`/`FIXME`/`HACK`/`XXX`/`WIP`/`DEPRECATED` 标记的函数、路由、组件，一律排除
- 标记出现在同一代码块（函数/类/路由定义）内，该功能整体排除
- `@beta`/`@experimental` 标记的功能可标注为"即将推出"，但不能作为当前可用功能

#### 3. 条件禁用的功能

通过特性开关（Feature Flag）或环境变量控制的功能，需要判断其默认状态：

```bash
# 检测特性开关（仅代码文件）
grep -rn -E '(featureFlag|feature_flag|FEATURE_|isEnabled|isDisabled|featureToggle)' --include="*.ts" --include="*.js" --include="*.py" --include="*.java"

# 检测环境变量控制的功能
grep -rn -E 'process\.env\.\w+|os\.environ\.get|System\.getenv' --include="*.ts" --include="*.js" --include="*.py" --include="*.java"
```

**处理规则：**
- 如果功能被 `if (false)`、`if (0)` 等恒假条件包裹，一律排除
- 如果功能被特性开关控制且默认关闭（`enabled: false`、`default: false`），排除
- 如果功能被特性开关控制且默认开启，可以保留，但需标注"可能因配置而异"

#### 4. 空实现/桩代码

```bash
# 检测空实现（仅代码文件）
grep -rn -E '(pass\s*$|not implemented|raise NotImplementedError|throw new Error.*not implemented)' --include="*.py" --include="*.js" --include="*.ts" --include="*.java"
```

**处理规则：**
- 函数体只有 `pass`、`NotImplementedError`、`// TODO` 等空实现，一律排除
- 只返回 mock 数据的接口，排除

### 功能可用性验证流程

对每个提取到的功能，按以下流程验证：

```
提取功能 → 是否被注释？ → 是 → 排除
              ↓ 否
         是否有 TODO/FIXME/DEPRECATED 标记？ → 是 → 排除
              ↓ 否
         是否被特性开关默认禁用？ → 是 → 排除
              ↓ 否
         是否为空实现/桩代码？ → 是 → 排除
              ↓ 否
         是否有对应的活跃路由/API？ → 否 → 排除
              ↓ 是
         纳入产品资料 ✅
```

## 约束三：甄别开源衍生代码

**⚠️ 项目很可能基于开源框架修改，必须区分"框架自带能力"和"项目实际开发的能力"，避免将开源框架的通用功能当作产品特色。**

### 开源框架识别

通过代码中的 import 来源识别项目使用的开源框架：

| 框架类型 | 识别信号 | 框架自带功能（不应作为产品特色） |
|----------|----------|-------------------------------|
| 后台管理模板 | `import.*from.*vue-element-admin\|ant-design-pro\|ng-alain\|blazor-admin` | 登录、菜单、权限、主题切换、多标签页、面包屑 |
| 低代码平台 | `import.*from.*amis\|formily\|form-generator\|drag-form` | 表单设计器、表格生成、图表配置 |
| CMS系统 | `import.*from.*strapi\|wordpress\|ghost\|headless-cms` | 内容管理、文章发布、媒体库、SEO |
| 电商框架 | `import.*from.*mall\|shopify\|magento\|woocommerce` | 商品管理、购物车、订单、支付 |
| ERP框架 | `import.*from.*odoo\|erpnext\|metabase` | 财务、库存、采购、销售 |
| 工作流引擎 | `import.*from.*activiti\|camunda\|flowable\|n8n` | 流程设计、审批、任务分配 |
| API网关 | `import.*from.*kong\|apisix\|tyk` | 路由、限流、认证、日志 |

```bash
# 检测开源框架引用（仅代码文件）
grep -rn -E 'from\s+["\']@(ant-design|vue-element|umi|max|nicegui)' --include="*.ts" --include="*.js" --include="*.tsx" --include="*.vue"
```

### 甄别原则

1. **框架基础功能 vs 项目定制功能**

| 判断维度 | 框架基础功能（排除） | 项目定制功能（保留） |
|----------|---------------------|---------------------|
| 代码位置 | `node_modules/`、`vendor/` | 项目的 `src/`、`app/`、`lib/` |
| 文件归属 | 第三方包内的代码 | 项目自有目录中的代码 |
| 修改痕迹 | 未被修改的原始代码 | 有明显业务逻辑修改的代码 |
| 业务关联 | 通用功能（登录、列表、表单） | 特定业务功能（如"合同审批"、"库存预警"） |

2. **如何判断代码是否被项目实际修改/使用**

```bash
# 检查某个组件是否被项目代码实际引用（而非仅存在于node_modules）
# 步骤1：找到组件名
# 步骤2：在项目自有代码中搜索引用
grep -rn "YourComponent" --include="*.tsx" --include="*.vue" --include="*.ts" --include="*.js" | grep -v "node_modules"

# 检查路由是否实际注册（而非仅定义）
grep -rn "use\|registerRoute\|addRoute" --include="*.ts" --include="*.js" | grep -v "node_modules"
```

3. **处理规则**

- **直接排除**：仅存在于 `node_modules/`、`vendor/` 等第三方目录中的功能
- **直接排除**：框架自带的通用基础功能（登录页、404页、通用列表页模板）
- **需要甄别**：框架组件被项目代码引用时——如果只是简单使用（`<Table>`、`<Form>`），属于框架能力；如果有大量业务逻辑包裹（自定义列渲染、复杂表单校验、业务联动），属于项目定制
- **保留**：项目自有目录中完全自主编写的业务代码

### 甄别流程

```
提取功能 → 是否来自第三方目录？ → 是 → 排除
              ↓ 否
         是否为框架通用基础功能？ → 是 → 排除
              ↓ 否
         项目代码中是否有实际引用？ → 否 → 排除
              ↓ 是
         是否包含项目定制的业务逻辑？ → 否 → 排除（仅为框架组件的简单调用）
              ↓ 是
         纳入产品资料 ✅
```

## 约束四：后端能力必须对应前端入口

**⚠️ 产品资料面向的是用户可感知、可操作的功能。如果后端实现了某个能力但前端没有暴露给用户，则该能力不属于产品功能，不应写入产品介绍。**

### 核心原则

> **没有前端入口的后端功能 = 内部能力，不是产品功能。**

用户无法看到、无法操作的后端能力（如定时任务、内部数据同步、系统健康检查、管理命令等），即使代码完整实现，也不应出现在面向客户的产品资料中。

### 前后端对应验证方法

对每个后端功能（API端点/服务/模型），必须验证其是否有前端入口：

```bash
# 步骤1：获取后端API路径列表（仅代码文件）
# 例如提取到：/api/users, /api/orders, /api/internal/sync, /api/health

# 步骤2：在前端代码中搜索该API是否被调用
grep -rn "api/users\|/users" --include="*.ts" --include="*.tsx" --include="*.vue" --include="*.js" --include="*.jsx" | grep -v "node_modules"

# 步骤3：检查是否有对应的前端页面/组件
grep -rn "user\|User" --include="*.tsx" --include="*.vue" | grep -iE "page|view|component|route"
```

### 必须排除的后端能力（无前端入口）

| 类型 | 典型信号 | 示例 |
|------|---------|------|
| 内部API | 路径含 `internal/`、`admin/`、`system/`、`cron/` | `/api/internal/data-sync` |
| 健康检查 | 路径含 `health`、`ping`、`status` | `/api/health` |
| 定时任务 | 代码含 `@Scheduled`、`cron`、`setInterval`、`celery` | `syncDataTask()` |
| 消息队列消费 | 代码含 `@Consumer`、`@Subscribe`、`on_message` | `handleOrderEvent()` |
| 数据迁移 | 代码含 `migration`、`seed`、`fixture` | `migrate_v2_to_v3()` |
| 系统管理命令 | 代码含 `manage.py`、`artisan`、`CLI` | `python manage.py cleanup` |
| Webhook回调 | 路径含 `webhook`、`callback`、`notify` | `/api/payment/callback` |
| 中间件/拦截器 | 代码含 `middleware`、`interceptor`、`filter` | `AuthMiddleware` |
| 后台Job | 代码含 `BackgroundJob`、`JobQueue`、`Worker` | `SendEmailJob` |

### 验证流程

```
提取后端功能 → 是否有前端API调用？ → 否 → 排除（内部能力）
                    ↓ 是
               是否有对应页面/组件？ → 否 → 排除（仅API无界面）
                    ↓ 是
               页面/组件是否被路由注册？ → 否 → 排除（未上线页面）
                    ↓ 是
               纳入产品资料 ✅
```

### 特殊情况处理

| 场景 | 处理方式 |
|------|---------|
| 后端API存在但前端暂未对接 | 排除，属于"后端已就绪、前端未开发" |
| 后端API仅被其他后端服务调用（微服务间通信） | 排除，属于内部架构能力 |
| 后端管理API（超级管理员专用） | 可保留，但标注为"管理员功能" |
| 后端导出/下载API（前端有下载按钮触发） | 保留，前端按钮即为入口 |
| 后端WebSocket推送（前端有实时数据展示） | 保留，前端实时展示即为入口 |
| 后端文件上传API（前端有上传组件） | 保留，前端上传组件即为入口 |

## 工作流程

### 核心分析逻辑：前端驱动，后端追踪

```
                    ┌──────────────────────────────────────┐
                    │  第一步：扫描前端路由                    │
                    │  提取所有用户可访问的页面/路径           │
                    └──────────────┬───────────────────────┘
                                   ↓
                    ┌──────────────────────────────────────┐
                    │  第二步：逐页面阅读前端代码              │
                    │  提取每个页面调用了哪些后端API           │
                    └──────────────┬───────────────────────┘
                                   ↓
                    ┌──────────────────────────────────────┐
                    │  第三步：仅阅读被调用的后端API代码       │
                    │  提取API的业务逻辑、数据模型            │
                    └──────────────┬───────────────────────┘
                                   ↓
                    ┌──────────────────────────────────────┐
                    │  第四步：整合为完整功能描述              │
                    │  前端交互体验 + 后端业务能力 → 功能卡片  │
                    └──────────────┬───────────────────────┘
                                   ↓
                    ┌──────────────────────────────────────┐
                    │  第五步：过滤 + 甄别 + 生成文档          │
                    │  应用所有约束规则，输出产品资料          │
                    └──────────────────────────────────────┘
```

**关键原则：后端代码只在前端调用的范围内阅读，前端没有调用的后端代码一律不读、不写。**

---

### 第一步：扫描前端路由（确定功能边界）

前端路由 = 用户能看到的全部功能边界。只有出现在路由中的页面，才有可能成为产品功能。

```bash
# 查找路由配置文件（仅代码文件）
find . -name "*route*" -o -name "*router*" | grep -E '\.(tsx?|jsx?|vue)$'

# 提取所有注册的路由路径
grep -rn -E 'path\s*[:=]\s*["\']([^"\']+)["\']' --include="*.tsx" --include="*.jsx" --include="*.vue" --include="*.ts" --include="*.js"
```

**输出：页面清单**
```
/                    → 首页/仪表盘
/users               → 用户管理
/orders              → 订单管理
/settings           → 系统设置
...
```

**注意：**
- 被注释掉的路由 → 排除
- 包含 TODO/FIXME 的路由文件 → 排除
- 框架通用路由（login、404） → 排除

---

### 第二步：逐页面追踪前端API调用（建立前后端关联）

对第一步得到的每个页面，阅读其前端代码，提取该页面调用了哪些后端API。

```bash
# 以"订单管理"页面为例，查找该页面调用的API
# 1. 找到页面文件
find . -path "*/pages/*order*" -o -path "*/views/*order*" | grep -E '\.(tsx?|jsx?|vue)$'

# 2. 在页面代码中提取API调用
grep -n -E '(axios|fetch|request|http|api)\.\s*(get|post|put|delete|patch)' <页面文件>
grep -n -E '["\']/(api|v1|v2)/[^"\']+["\']' <页面文件>
```

**输出：页面 → API关联表**
```
订单管理页面:
  - GET    /api/orders          → 查询订单列表
  - POST   /api/orders          → 创建订单
  - GET    /api/orders/:id      → 订单详情
  - PUT    /api/orders/:id      → 更新订单
  - DELETE /api/orders/:id      → 删除订单
  - POST   /api/orders/export   → 导出订单

用户管理页面:
  - GET    /api/users           → 查询用户列表
  - POST   /api/users           → 创建用户
  ...
```

**注意：**
- 仅提取页面代码中实际发起的API调用，不提取未使用的API定义
- 前端代码中有调用但被条件包裹（`if (false)`）的API → 排除

---

### 第三步：顺藤摸瓜阅读后端代码（仅阅读被调用的API）

**只阅读第二步中提取到的API对应的后端代码**，不浏览其他后端文件。

```bash
# 以 GET /api/orders 为例，在后端代码中定位该API的实现
# Node.js/Express
grep -rn 'router\.get\s*\(\s*["\'].*orders' --include="*.js" --include="*.ts"

# Java/Spring
grep -rn '@GetMapping.*orders' --include="*.java"

# Python/FastAPI
grep -rn '@app\.get.*orders\|@router\.get.*orders' --include="*.py"
```

**对每个API，提取以下信息：**

| 提取项 | 从哪里提取 | 转化为产品信息 |
|--------|-----------|---------------|
| API处理函数 | Controller/Router文件 | 业务能力描述 |
| 调用的Service层 | import语句追踪 | 核心业务逻辑 |
| 操作的数据模型 | Service中的ORM调用 | 管理的数据对象 |
| 参数校验规则 | 请求参数定义 | 数据要求 |
| 权限控制 | 装饰器/注解/中间件 | 安全特性 |

**追踪链路示例：**
```
GET /api/orders
  → OrderController.getAllOrders()
    → OrderService.queryOrders()
      → OrderModel.find()        → 数据模型：Order
      → OrderService.applyFilter() → 业务逻辑：多条件筛选
      → OrderService.calculateTotal() → 业务逻辑：自动汇总
```

**注意：**
- 只追踪API调用链路上的代码，不阅读无关的Service/Model
- 如果后端API文件包含 TODO/FIXME → 该API对应的整个功能排除
- 如果后端API是空实现 → 该功能排除

## 产出文档资料准则

**⚠️ 以下4条准则是生成最终文档时必须遵守的撰写规范，确保产出的产品资料专业、可信、可直接发给客户。**

### 准则一：专业化语言

使用行业规范词汇替代口语化表达，提升文档的专业感和可信度。

**词汇升级对照表：**

| ❌ 口语化表达 | ✅ 专业化表达 |
|-------------|-------------|
| 很快 | 毫秒级响应 |
| 不会崩 | 高可用保障 |
| 很安全 | 全链路隔离 |
| 能扛住大量用户 | 弹性伸缩架构 |
| 数据不会丢 | 数据持久化保障 |
| 操作简单 | 极简交互体验 |
| 什么都能接 | 全生态兼容 |
| 管得很细 | 精细化管控 |
| 看得清楚 | 可视化洞察 |
| 随时能用 | 即开即用 |
| 不会出错 | 智能校验机制 |
| 自动处理 | 全自动化流转 |
| 谁都能用 | 零门槛上手 |

**高频专业词汇库：**

| 场景 | 推荐用词 |
|------|---------|
| 性能 | 毫秒级响应、亚秒级处理、高性能吞吐、低延迟 |
| 稳定性 | 高可用保障、容灾备份、故障自愈、服务降级 |
| 安全 | 全链路隔离、端到端加密、零信任架构、多层防护 |
| 架构 | 企业级架构、弹性伸缩、微服务解耦、分布式部署 |
| 数据 | 数据资产化、智能洞察、全量溯源、实时同步 |
| 体验 | 极简交互、沉浸式体验、场景化引导、个性化适配 |
| 管理 | 精细化管控、全生命周期管理、智能预警、闭环运营 |

### 准则二：逻辑闭环

每一项技术特征必须遵循 **[技术实现 → 业务功能 → 客户效益]** 的三段式逻辑，形成完整的说服链。

**结构模板：**

```
[技术实现]：采用 XXX 技术/方案
    ↓
[业务功能]：实现了 XXX 能力
    ↓
[客户效益]：帮助客户 XXX（降本/增效/控险/提质）
```

**正确示例：**

| 技术实现 | 业务功能 | 客户效益 |
|----------|----------|----------|
| 采用分布式缓存与读写分离架构 | 支持万级并发实时数据查询 | 业务高峰期零卡顿，用户体验不降级 |
| 基于RBAC + 数据级权限模型 | 实现部门、角色、人员三级精细化管控 | 杜绝越权操作，满足合规审计要求 |
| 引入消息队列异步处理机制 | 批量任务后台自动执行 | 人工操作时间减少80%，释放人力成本 |
| 多活部署 + 自动故障转移 | 服务可用性达99.99% | 业务连续性保障，避免停机损失 |

**错误示例（缺少闭环）：**

| ❌ 不完整描述 | 缺失环节 |
|-------------|---------|
| 系统采用Redis缓存 | 缺少业务功能和客户效益 |
| 支持批量导入导出 | 缺少技术实现和客户效益 |
| 帮助企业降本增效 | 缺少技术实现和业务功能 |

**检查方法：** 对每段描述提问"所以呢？"——如果答不上来客户获得了什么具体好处，说明逻辑链断裂。

### 准则三：去工程化

**绝对不要在产品资料中出现任何工程化细节。** 客户关心的是"能做什么"，而非"怎么做的"。

**禁止出现的内容：**

| 类别 | ❌ 禁止示例 | ✅ 替代方案 |
|------|------------|------------|
| 代码语法 | `const user = await User.findById(id)` | "支持按条件快速检索用户信息" |
| 类名/函数名 | `UserService.createOrder()` | "一键创建业务订单" |
| 数据库表名 | `SELECT * FROM t_order_detail` | "订单明细全维度查询" |
| API路径 | `POST /api/v1/users/login` | "安全快捷的用户认证" |
| 变量名 | `const isAuthenticated = true` | "多重身份验证机制" |
| 框架版本 | "基于 Vue 3.4 + Vite 5.x" | "采用新一代前端技术架构" |
| 包名/模块名 | `import { Button } from 'antd'` | "标准化UI组件体系" |
| 端口号 | "服务运行在 8080 端口" | "支持私有化部署" |
| 文件路径 | `src/views/dashboard/index.vue` | "管理驾驶舱" |
| 配置项 | `timeout: 3000ms` | "毫秒级响应" |

**去工程化转化原则：**

1. **类名 → 功能名**：`OrderController` → "订单管理中心"
2. **函数名 → 操作名**：`calculateDiscount()` → "智能折扣计算"
3. **变量名 → 业务概念**：`isVip` → "会员等级识别"
4. **技术栈 → 能力描述**：`Elasticsearch` → "全文智能检索引擎"
5. **配置参数 → 体验描述**：`pageSize: 20` → "高效分页加载"

### 准则四：视觉化导向

在描述前端代码时，**不要描述代码结构，而是转化为"用户交互体验"和"数字化驾驶舱"的描述**。

## 去AI味文风准则

**⚠️ 产出内容必须像一个行业专家在向客户面对面讲述，而不是AI生成的模板化内容。**

### 核心原则：说人话，讲事实，给方案

> 想象你是一位有10年行业经验的资深顾问，正在向客户的决策层做产品汇报。你的语气应该是自信、务实、专业的，而不是机械、堆砌、浮夸的。

### 六条去AI味规则

#### 规则一：禁止AI高频词

以下词汇是AI生成内容的典型标志，**一律禁止使用**：

| ❌ 禁止 | ✅ 替代 |
|---------|---------|
| 赋能 | 支持、助力、帮助 |
| 旨在 | 目标是、专注于 |
| 打造 | 构建、建设、提供 |
| 引领 | 推动、促进 |
| 全方位 | 多维度、全面 |
| 深度 | 专项、系统化 |
| 一站式 | 集成化、统一 |
| 无缝 | 顺畅、高效 |
| 智能（滥用） | 自动化、精准（具体说明什么智能） |
| 创新（滥用） | 新型、改进型（具体说明什么创新） |
| 依托...优势 | 基于...经验 |
| 致力于 | 专注于、聚焦于 |
| 助力...实现 | 帮助...完成 |
| 推动...发展 | 促进...提升 |
| 构建了...体系 | 建立了...机制 |
| 实现了...突破 | 达到了...水平 |

#### 规则二：用具体事实替代空泛描述

| ❌ 空泛（AI味） | ✅ 具体（专家味） |
|----------------|-------------------|
| 提供全方位的智能化解决方案 | 支持从数据采集到分析决策的4个核心环节 |
| 具有强大的数据处理能力 | 单次可处理10万条记录，查询响应时间低于2秒 |
| 显著提升工作效率 | 公文起草时间从2天缩短至30分钟 |
| 满足多样化需求 | 支持Word、Excel、PDF等6种格式导入 |
| 广泛应用于各行业 | 已在政务、金融、教育3个行业落地应用 |

#### 规则三：句式要短、要直接

**AI味的句式特征：**
- 长句嵌套："通过...的方式，实现了...的目标，从而...的效果"
- 排比堆砌："不仅...而且...同时...此外..."
- 被动语态过多："被广泛应用于"、"被证明是"

**专家味的句式特征：**
- 短句为主，一事一议
- 主语明确，动作清晰
- 适当使用主动语态

```
❌ AI味：
"通过构建'通用大模型+场景化专家模型+专业知识库'的AI能力内核，
致力于为各行业提供高效、智能、安全的场景化智能体应用服务，
助力政务、金融、教育等行业客户实现智能化转型与升级，
推动业务效率的大幅提升与创新发展。"

✅ 专家味：
"平台以通用大模型为基础，结合行业专家模型和专业知识库，
为政务、金融、教育等行业提供场景化的智能应用。
目前已在多个项目中落地，公文起草效率提升80%以上。"
```

#### 规则四：功能描述用"简介+场景"结构

参考手册中每个功能的写法：

```
1. 智能体简介：[2-3句话，说清楚这是什么、能做什么、核心特点]

   - [行业A]：[具体角色]在做[具体事情]时，[功能]可以[具体效果]

"2. 安全可靠：采用多重安全机制，全方位保障数据安全"

✅ 专家味：
"1. 模型准确率保持在95%以上，远超通用大模型。通过精细调校，
   大幅提升准确度与召回率，有效抑制模型幻觉，减少错误与遗漏。"
"2. 支持全信创方案，从硬件到模型均支持国产组件与技术，
   满足关键行业对自主可控的要求。"
```

#### 规则六：案例描述要有角色、有动作、有结果

```
❌ AI味：
"在某政务项目中，本产品发挥了重要作用，取得了显著成效。"

✅ 专家味：
"浙江省某市政府部门引入AI智能体平台后，借助公文写作智能体，
公务人员只需输入议题和要求，便能快速生成规范的议案、报告等公文初稿，
大大缩短了公文起草时间，且内容质量和格式规范性均有保障。"
```

### 自检方法

写完每段内容后，问自己三个问题：
1. **删掉产品名后，这段话是否仍然成立？** 如果不成立，说明在自吹自擂而非描述事实
2. **把这段话念给非技术人员听，能听懂吗？** 如果听不懂，说明太技术化或太抽象
3. **这段话如果是你当面跟客户说的，你会觉得尴尬吗？** 如果会，说明太浮夸或太模板化

## 最佳实践

### 代码分析原则

1. **仅基于源代码** - 所有信息必须从源代码文件中提取，绝不参考 README、配置文件、文档等非代码文件
2. **仅包含上线功能** - 排除被注释、TODO/FIXME/DEPRECATED 标记、条件禁用、空实现的功能代码
3. **甄别开源衍生** - 区分框架自带能力和项目定制能力，不将开源框架通用功能当作产品特色
4. **后端必须对应前端** - 没有前端入口的后端能力属于内部能力，不纳入产品资料
5. **从用户视角出发** - 关注代码实现的功能，而非实现细节
6. **提取业务价值** - 每个功能都要说明解决什么问题
7. **使用产品语言** - 避免技术术语，使用客户易懂的表达
8. **突出差异化** - 找出产品独特的技术亮点和功能特色

### 文档生成原则

1. **结构清晰** - 使用层次分明的章节结构
2. **图文并茂** - 适当添加架构图、流程图
3. **重点突出** - 核心功能放在显眼位置
4. **专业美观** - 统一的排版风格和配色方案

### 质量检查清单

- [ ] 所有功能模块都已从代码中提取
- [ ] 未读取任何非代码文件（README、package.json、配置文件等）
- [ ] 已排除被注释掉的功能代码
- [ ] 已排除包含 TODO/FIXME/DEPRECATED 标记的功能
- [ ] 已排除被特性开关默认禁用的功能
- [ ] 已排除空实现/桩代码
- [ ] 已排除仅存在于 node_modules/vendor 中的框架自带功能
- [ ] 已区分框架通用功能与项目定制功能
- [ ] 后端功能均有对应的前端入口（页面/组件/API调用）
- [ ] 已排除无前端入口的内部后端能力（定时任务、健康检查、消息消费等）
- [ ] **专业化语言**：无口语化表达，使用了行业规范词汇
- [ ] **逻辑闭环**：每项技术特征都包含 [技术实现→业务功能→客户效益] 三段
- [ ] **去工程化**：无代码语法、类名、函数名、数据库表名、API路径等工程细节
- [ ] **视觉化导向**：前端功能描述已转化为用户交互体验和数字化驾驶舱视角
- [ ] **文档结构**：遵循"概述→优势→功能→案例→服务"的标准结构
- [ ] **去AI味**：无"赋能、旨在、打造、引领、全方位、一站式"等AI高频词
- [ ] **专家口吻**：句式短直接、用事实说话、功能有简介有场景、案例有角色有结果
- [ ] 文档结构完整清晰
- [ ] 排版格式专业美观
- [ ] 无敏感信息泄露（API密钥、内部IP、密码等）

## 常见问题

### Q: 如何处理复杂的技术实现？

**A:** 只提取用户可感知的功能，内部实现细节转化为"高性能"、"高可靠"等产品特性描述。

### Q: 如何处理未完成的功能？

**A:** 严格排除。任何包含 `TODO`、`FIXME`、`WIP`、`DEPRECATED`、`@deprecated` 标记的代码块，以及被注释掉的代码、空实现（`pass`、`NotImplementedError`），一律不纳入产品资料。

### Q: 如何判断功能是否实际上线？

**A:** 按验证流程逐项检查：是否被注释？是否有开发中标记？是否被特性开关禁用？是否为空实现？是否有对应的活跃路由或API注册？只有全部通过才纳入。

### Q: 项目基于开源框架修改，如何避免把框架功能当作产品功能？

**A:** 三步甄别：(1) 排除 `node_modules/`/`vendor/` 中的代码；(2) 排除框架自带的通用基础功能（登录、菜单、列表模板等）；(3) 只保留项目自有目录中包含业务定制逻辑的代码。判断标准：如果去掉框架名，这个功能描述是否仍然成立？如果不成立（如"基于Ant Design的表格"），说明是框架能力而非产品能力。

### Q: 如何保护代码安全？

**A:** 
- 不要在文档中包含具体的API密钥、密码
- 不要暴露内部服务器地址
- 不要详细描述安全机制的实现细节
