# 量潮数字资产云

接下来我打算尝试开发 provider 和 studio 的 0.0.1。如果需要的话，命令行也同时发布 0.0.2如果这些都通过的话，我们就，给主仓库发布 0.0.1。

这个版本的主要目标是实现本地的资产管理选择这个场景的主要原因是当前还没有很完善的账号系统，但是我们又需能够集中进行治理，而不仅仅是在本地仓库进行治理，那么我们就需要一个管理数据的注册中心所以我们计划模仿 openco 在本地运转的架构，搭建一个数字资产 1 的架构，然后后台放一个 provider，然后前台同时允许 studio 和 CL 去访问这个 provider，然后这个 provider 去管理需要集中自己的信息。这里需要验证的核心范式是，能不能把我的私人的第二大脑和我的公开第二大脑进行对接。这个回答的问题是在同一类型的资产之下，比如说同样都是本地文件和 git 的这种模式之下，我们能不能去从一个框架内的结构给先转移到另外一个框架？这个回答的是跨人员跨组织之间的信息流转问题，是数字资产治理的核心目的之一。

我设想的场景是这样子的，就是我们能够在本地看得到我有这么多资产被注册了，就是我的仓库资产有一个固定的文件夹，扫描这个文件夹就能够获得所有的信息，然后呢，我也有一些现成的资产清单，然后呢，这些资产清单可以辅助。然后呢，我要进行的日常操作是，比如说我在私人记忆仓库里边放了一个上下文资料，我希望把它转移到公开的文档里面，然后这本质上其实就是一个在本地进行的文件迁移操作，然后呢，它只需要能够调用文件系统的那个权限就可以。

然后这里的一个难点是对规则的显性化，就是从哪一个目录到哪个目录目录完成这个转移提交的操作。然后就是这个过程可以自动完成，然后然后提交的过程就是云端提交这个过程还可以继续有人来维护这样的话，我们验证核心的逻辑是可靠的就好。

通过这个操作呢，我们还有可能会得到一些可以进行，那个存放成工作简介的一些或者是工作案例的一些重要信息，就是我们可以用这些信息来去规范我们系统内的活动，具体怎么存放，看看到时候做出来是什么东西。总体来讲，因为在早期的话它的。案例价值可能会大于它的信息价值，所以可能更多的是案例。
